# TASK-052 — Make the update-check usable: land v0.8.1 metadata + publish the first GitHub Release

## Why

TASK-050 shipped the U-lite update check and TASK-051 corrected the repository owner
(`kaikkonen4` → `kaikkone4`, merged in PR #39, `origin/main` @ `a3bd398`). The owner fix is now on
`main`, so the GitHub targets point at the real repository. But the feature still cannot resolve to
a real state, for two independent reasons confirmed against the live repo:

1. **No GitHub Release exists.** `gh release list --repo kaikkone4/vire` returns empty, so
   `https://api.github.com/repos/kaikkone4/vire/releases/latest` returns **404**. Every
   "Check for updates" therefore fail-softs to `Unknown` even though the URL is now correct.
2. **`main` still ships `0.8.0` metadata.** The v0.8.1 version bump
   (`src-tauri/Cargo.toml`, `src-tauri/tauri.conf.json`) was prepared by TASK-051 SW-6 as local
   commit `3beece9` but **never merged** — PR #39 carried only the URL fix (`c0d2350`). The TASK-051
   post-merge code-to-spec verification recorded this as **DIVERGED**
   (`openspec/changes/task-051-fix-update-release-url/code-to-spec.md`). The `v0.8.1` tag was also
   never created (the signed-tag step was blocked on an interactive passphrase).

The fix is small and operational: reconcile the app's compiled version metadata to `0.8.1` on
`main`, then **publish a single non-draft GitHub Release tagged `v0.8.1`** so the latest-release
endpoint returns `200`. No update-check *logic* changes — `src-tauri/src/update_check/mod.rs`
already resolves correctly against a real release.

## What changes

Two ordered parts in the **Vire desktop app** component, with a strict ownership boundary between
them (the code change goes through the normal SW gates; the public Release is published by the
Release Manager *after* merge — see `arch-review.md`):

**Part A — Code change (SW-2 → gates → merge), branch off `origin/main`:**

- `src-tauri/Cargo.toml` — `version = "0.8.0"` → `"0.8.1"`.
- `src-tauri/tauri.conf.json` — `"version": "0.8.0"` → `"0.8.1"`.
- `src-tauri/Cargo.lock` — the `vire` package `version` entry follows to `0.8.1` (regenerated by
  `cargo build`; commit the lockfile delta).
- `RELEASE.md` (repo root) — prepend a `## v0.8.1` changelog entry rolling up the update-check work
  (TASK-050 check + TASK-051 owner fix + TASK-052 release publication).

**Part B — Release publication (SW-6, after merge to `main`):**

- Publish a GitHub Release tagged **`v0.8.1`**, **full release** (NOT `--draft`, NOT
  `--prerelease`), targeting the merged `main` commit that carries `0.8.1`. Use
  `gh release create v0.8.1 --target <merged-sha>` so GitHub creates the tag **server-side**,
  sidestepping the local GPG-passphrase blocker that stalled TASK-051's signed tag.

## What does NOT change

- **No update-check logic change.** `update_check/mod.rs`, the IPC commands, the bounded fail-soft
  client, the scoped opener capability, and the canonical-owner regression guards are untouched.
- **No `tauri-plugin-updater`, no download, no install, no code signing, no notarization.** The
  in-app updater (TASK-042 territory) stays out of scope.
- **No binary/installer build, sign, or upload.** Attaching `.dmg`/`.app` assets to the Release is
  **out of scope** and explicitly deferred — the latest-release endpoint, semver comparison, and
  "Open Releases" all function from a notes-only Release. See `arch-review.md` "Known limitation".
- **No schema, migration, IPC-storage, CSP `connect-src`, capture, classifier, or Langfuse change.**
- No manual edits to generated files (`src-tauri/gen/schemas/capabilities.json`,
  `src-tauri/target/**`); they regenerate at build.
- `package.json` has no `version` field and gains none — the canonical version source stays
  `Cargo.toml` / `tauri.conf.json` (in-app value via `CARGO_PKG_VERSION`).

## Impact

- **Makes the headline TASK-050 behaviour real for the first time.** Once `v0.8.1` is published,
  the currently distributed `0.8.0` build resolves "Check for updates" to **UpdateAvailable
  (0.8.0 → 0.8.1)** with a real `release_url`; a rebuilt `0.8.1` build resolves to **UpToDate**.
  "Open GitHub Releases" reaches a real release page.
- **Resolves the TASK-051 divergence** by landing the `0.8.1` metadata on `main` through a clean,
  reviewable diff (rather than depending on the unmerged, tag-blocked local commits).
- **No security-boundary change.** Egress host set, capability type, CSP, and client posture are
  identical to TASK-050/051; SEC-002 is unaffected. Publishing a public GitHub Release is an
  outward-facing action owned by SW-6 (see `arch-review.md`).
- **Component scope:** one component (the desktop app) plus a release-ops action. No cross-boundary
  work; **no task split required.**
